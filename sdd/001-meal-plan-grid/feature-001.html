Feature 001

Weekly Meal Plan Grid

A 7-day meal planning grid for a single household. Meals are assigned to configurable meal-type rows, ingredient lists are generated asynchronously by a local Ollama model, and toddler-appropriate constraints are enforced at slot confirmation.

Status: In development
Tasks: T001 – T016
Depends on: None
Unlocks: Feature 002, 003

01 Acceptance Criteria

AC-001

Slot saves immediately on confirm. Loading indicator appears. Toddler-appropriate question asked.

AC-002

Ingredients populate asynchronously without blocking other slots.

AC-003

Ollama unavailable: slot saves with empty ingredients. Manual add available.

AC-004

Past days are fully read-only. No edit controls rendered.

AC-005

Week navigation via ‹ / › on the date range display. Disabled at boundaries.

AC-006

Toddler home day flagging checks existing slots for conflicts before saving.

AC-007

Sunday is always the first column. Week boundary at 00:00 home timezone.

AC-008

Editing meal name discards ingredients and re-runs Ollama.

03 Data Model

Meal types are stored as MealTypeConfig records — not hardcoded as an enum. This lets the household configure their own meal structure (e.g. add a "Supper" row) without any code changes. Three defaults are seeded on first run: Breakfast, Lunch, Dinner.

Key design decisions

Dates as strings. date is stored as YYYY-MM-DD text, not DateTime. This avoids timezone drift when reading back from SQLite. The home timezone is applied at the API layer.

Soft-delete for meal types. MealTypeConfig.isActive hides types from the grid without orphaning existing slot data.

@@unique([date, mealTypeConfigId]). One slot per meal type per day, enforced at the database level. Duplicate attempts return 409.

Ingredient approval. Ingredient.approved tracks whether the user has reviewed each ingredient before it flows to the shopping list in Feature 003.

06 Sequence: Saving a Meal Slot

The slot is persisted before Ollama responds. The HTTP response is 201. Ingredient generation runs in a background task (scheduleIngredientGeneration via Next.js after()). MealPlanGrid SWR polls every 3s while any slot is PENDING.

Sequence Diagram — Save Meal Slot + Async Ingredient Generation

    participant UI as MealSlotEditing

    participant OL as Ollama

    U->>UI: Types meal name, confirms
    UI->>API: POST meal slot JSON
    API->>DB: Reject past day with 403
    API->>DB: Reject duplicate with 409
    API->>OL: GET tags reachability check
    alt Ollama reachable
        API->>DB: INSERT slot as PENDING
    else Ollama unavailable
        API->>DB: INSERT slot as EMPTY
    end
    API-->>UI: 201 slot JSON
    UI->>Grid: onSaved revalidates SWR
    UI-->>U: Filled tile with spinner

    Note over API,BG: scheduleIngredientGeneration after response
    API->>BG: enqueue generateIngredients
    BG->>OL: POST api generate
    alt success
        OL-->>BG: comma-separated ingredients
        BG->>DB: UPDATE READY and insert rows
    else timeout or error
        BG->>DB: UPDATE FAILED
    end

    loop Every 3s while slot is PENDING
        Grid->>API: GET meal plan
        API->>DB: Fetch week data
        API-->>Grid: Updated days payload
        Grid-->>U: Spinner becomes ingredient list
    end

07 Sequence: Loading the Meal Grid

SWR key: /api/meal-plan?offset={n} (also supports ?week=YYYY-MM-DD for a Sunday). Polls every 3 seconds while any slot has ingredientsStatus === PENDING via getRefreshInterval(). Polling stops when no slots are pending.

    loop Every 3s while any slot is PENDING
        Grid->>API: GET meal-plan with offset
        API->>DB: Fetch week data
        API-->>Grid: Updated days payload
        Grid-->>U: Spinner becomes ingredient list when READY
    end

    Note over Grid: refreshInterval 0 when no PENDING slots

08 Key Constraints

Non-negotiable rules from the project constitution

RuleImplementation
Home timezone is the single source of truth date-fns-tz via lib/date.ts. new Date() never used for day boundaries.
Ollama never blocks the HTTP response void generateIngredients() — fire and forget after slot is saved.
Past days are read-only API returns 403 on write. UI renders no controls on past cells.
Week starts on Sunday startOfWeek(date, { weekStartsOn: 0 }) everywhere.
No hardcoded meal types DayColumn renders slots from the mealTypes[] API response, never from an enum.
No raw process.env outside config All env vars accessed via lib/config.ts typed exports.
